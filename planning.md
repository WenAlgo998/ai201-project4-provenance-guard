# Provenance Guard — Planning

Provenance Guard is a backend service that any creative-sharing platform (writing,
poetry, blogging) can call to estimate whether a submitted piece of text was written
by a human or generated by AI. It is deliberately **not** a verdict machine: it
reports a verdict _with_ a confidence score, surfaces a plain-language transparency
label, and gives creators a path to appeal a classification they believe is wrong.

The guiding principle throughout this design is an **asymmetry of harm**: on a
writing platform, falsely labeling a human's original work as "AI-generated" is far
more damaging than failing to catch a piece of AI text. A false positive can damage a
real creator's reputation and livelihood. Every design decision below — the wide
"uncertain" band, the human-biased thresholds, the hedged label language, the appeals
workflow — is shaped by that asymmetry.

## Architecture

### Components

| Component                                                      | Responsibility                                                                                                                     |
| -------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **Flask API**                                                  | Exposes the HTTP contract (`/submit`, `/appeal`, `/log`, …) and orchestrates the pipeline.                                         |
| **Rate limiter** (Flask-Limiter)                               | Protects `/submit` from flooding/abuse before any expensive work runs.                                                             |
| **Signal 1 — LLM classifier** (Groq `llama-3.3-70b-versatile`) | Holistic semantic/stylistic judgment → `P(AI)` in `[0,1]` + short rationale.                                                       |
| **Signal 2 — Stylometry** (pure Python)                        | Structural statistics (burstiness, vocabulary diversity, punctuation density) → `P(AI)` in `[0,1]`.                                |
| **Confidence scorer**                                          | Combines the two signal scores into one `combined_ai_score` and a `confidence` value, penalizing confidence when signals disagree. |
| **Label builder**                                              | Maps `(combined_ai_score, confidence)` to one of three plain-language transparency labels.                                         |
| **Audit log** (SQLite)                                         | Append-only structured record of every decision and every appeal, timestamped.                                                     |
| **Content store** (SQLite)                                     | Holds each submission's current status (`classified` → `under_review`).                                                            |

## Architecture narrative (the path of one piece of text)

A creator submits a poem via **`POST /submit`**. The request first passes the
**rate limiter**, which rejects it with `429` if the creator (keyed by IP/creator_id)
has exceeded the configured budget; this happens before any expensive analysis so an
attacker can't flood the LLM. The raw text is then sent to both detection signals.
**Signal 1 (LLM)** asks Groq's `llama-3.3-70b-versatile` to judge how AI-like the
writing reads and return a probability plus a one-line rationale. **Signal 2
(stylometry)** computes structural statistics in pure Python and maps them to a
probability — no network call, so it always runs even if Groq is unavailable. The
**confidence scorer** combines the two probabilities into a single `combined_ai_score`
and a `confidence` value, lowering confidence when the two signals disagree. The
**label builder** turns that pair into one of three plain-language transparency labels.
The full decision — verdict, confidence, both individual signal scores, and the label —
is written to the **audit log** and the submission is recorded in the **content store**
with status `classified`. Finally the API returns a structured JSON response containing
the attribution result, the confidence score, the individual signal scores, and the
exact label text the platform would show the reader.

If the creator believes the verdict is wrong, they call **`POST /appeal`** with their
reasoning. The **content store** flips the submission's status to `under_review`, and an
appeal entry is appended to the **audit log** directly alongside the original
classification, capturing the creator's reasoning and a timestamp. The response confirms
the new `under_review` status. (Automated re-classification is intentionally out of scope;
a human moderator resolves the review.)

## Detection signals

The two signals are chosen to be **genuinely independent**: one judges _meaning and
voice_, the other judges _structure and statistics_. Because they fail in different
ways, agreement between them is informative — it's the basis of the confidence score.

### Signal 1 — LLM semantic classifier (Groq `llama-3.3-70b-versatile`)

- **What it measures:** Holistic semantic and stylistic coherence — whether the text
  reads as authentically human. The model is prompted to weigh things like generic vs.
  specific detail, hedged/balanced phrasing, formulaic transitions ("Moreover,",
  "In conclusion,"), emotional texture, and idiosyncrasy, and to return `P(AI)` in
  `[0,1]` plus a short rationale.
- **Why this differs between human and AI:** AI prose tends to be fluent but
  _flavorless_ — evenly balanced, hedge-heavy, rich in connective tissue, and light on
  lived, specific detail. Human writing more often carries concrete specifics, uneven
  emphasis, deliberate rule-breaking, and a distinct voice.
- **What it can't capture (blind spot):** It has no ground truth and can be confidently
  wrong. It is **biased against non-native English speakers and formal/academic writers**,
  whose clean, uniform prose can read as "AI-like" — a major false-positive risk that
  directly motivates our human-biased thresholds. It can be gamed by prompting an AI to
  write "quirky," and it can't reliably detect lightly human-edited AI text.

### Signal 2 — Stylometric heuristics (pure Python, no external libraries)

- **What it measures:** Quantitative structural properties of the text:
    - **Sentence-length variance ("burstiness")** — the spread of sentence lengths.
    - **Type-token ratio** — vocabulary diversity (unique words ÷ total words).
    - **Punctuation density** — variety/rate of punctuation (dashes, semicolons, etc.).
    - **Average sentence complexity** — mean words per sentence / word length.
      These map to a single `stylo_ai_score` in `[0,1]`.
- **Why this differs between human and AI:** AI text is statistically _uniform_ — low
  sentence-length variance, middle-of-the-road vocabulary diversity, even punctuation.
  Human writing is _bursty_: short punchy sentences next to long winding ones, sharper
  swings in vocabulary, idiosyncratic punctuation.
- **What it can't capture (blind spot):** It is **meaning-blind** — it can't tell a
  uniform-but-genuinely-human technical writer from AI, and it is **unreliable on short
  texts** (a 2-sentence poem has too few data points for stable statistics). It is also
  easily gamed by deliberately varying sentence length, and its expectations are
  genre-dependent (poetry ≠ blog prose).

### Why the pairing is strong

One signal is _semantic_, the other _structural_. A clever adversary or an unusual
genuine writer can fool one but rarely both in the same direction. When they **agree**,
we can be confident; when they **disagree**, that disagreement is exactly the
"uncertainty" we want to surface to the user rather than hide.

## Confidence scoring & uncertainty representation

Each signal returns a probability that the text is AI-generated:
`llm_ai_score`, `stylo_ai_score` ∈ `[0,1]`.

**Combination (weighted, LLM-favored):**

```
combined_ai_score = 0.65 * llm_ai_score + 0.35 * stylo_ai_score
```

The LLM is weighted higher because semantic judgment is more informative than surface
statistics, but stylometry retains real weight as an independent check (and as the
fallback when Groq is unavailable).

**Agreement & confidence:**

```
agreement   = 1 - abs(llm_ai_score - stylo_ai_score)      # 1 = perfect agreement
raw_strength = abs(combined_ai_score - 0.5) * 2           # 0 at 0.5, 1 at the extremes
confidence   = raw_strength * agreement                    # disagreement pulls confidence down
```

`confidence` ∈ `[0,1]`. A score near 0.5 (the model is on the fence) **or** signals that
disagree both drive confidence toward 0 — so `0.51` and `0.95` produce genuinely
different labels, as required.

**What a confidence of 0.6 means to the system:** confidence is *how sure we are of the
verdict*, not the probability the text is AI. `confidence = 0.60` is our **decision
boundary**: it is the minimum certainty required to make a directional call at all.
Below `0.60` we refuse to commit and always show the `uncertain` label, regardless of
which side of `0.5` the score leans. So a `0.60` is a *just-barely-confident* result —
the weakest call we're willing to publish — while `0.95` is a near-certain one. This is
deliberate: the floor exists so that weak or conflicting evidence never produces an
accusation.

**Verdict bands (deliberately human-biased):**

| Condition                                                       | Verdict     | Label variant         |
| --------------------------------------------------------------- | ----------- | --------------------- |
| `combined_ai_score ≥ 0.65` **and** `confidence ≥ 0.60`          | `ai`        | High-confidence AI    |
| `combined_ai_score ≤ 0.35` **and** `confidence ≥ 0.60`          | `human`     | High-confidence human |
| everything else (`0.35 < score < 0.65`, or `confidence < 0.60`) | `uncertain` | Uncertain             |

The "uncertain" band is intentionally **wide**, and the bar to declare "AI" (`≥ 0.65`)
is higher than a naive `> 0.5` — both choices bias the system toward _not_ falsely
accusing a human. When in doubt, we say "uncertain," never "AI."

**How we'll validate the scores are meaningful (planned for Milestone 2/3):**

- Feed known-human text (hand-written, idiosyncratic) → expect `human`, high confidence.
- Feed obvious ChatGPT output (generic 5-paragraph essay) → expect `ai`, high confidence.
- Feed a borderline case (short, formal, human but uniform) → expect `uncertain`, low
  confidence — confirming the system hedges instead of falsely accusing.
- Feed a case where signals disagree (AI text edited to be bursty) → confirm confidence
  drops and the label becomes `uncertain`.

## Transparency label variants (verbatim text)

The label shown to readers uses plain language — no "score," "classifier," or "logit."
The wording differs by variant, and the AI variant is deliberately hedged.

| Variant                   | Title                   | Body text (verbatim)                                                                                                                                                                                                                   |
| ------------------------- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **High-confidence AI**    | 🤖 Likely AI-generated  | "Our analysis found strong signals that this text was generated by AI. Both how it reads and its writing patterns point this way. This is an automated estimate, not a certainty — the creator can appeal if they believe it's wrong." |
| **Uncertain**             | ❓ Origin unclear       | "We couldn't confidently tell whether this text was written by a person or generated by AI. Our signals were weak or disagreed with each other, so we're not making a call. Treat this as undetermined."                               |
| **High-confidence human** | ✍️ Likely human-written | "Our analysis found strong signals that a person wrote this text. How it reads and its writing patterns are both consistent with human authorship. This is an automated estimate, not a guarantee."                                    |

Each label also surfaces the numeric `confidence` for platforms that want it, but the
human-readable text stands on its own.

## API surface (the contract)

### `POST /submit`

Submit text for attribution analysis. **Rate limited.**

- **Accepts:** `{ "text": "<content>", "creator_id": "<optional>" }`
- **Returns `200`:**
    ```json
    {
        "content_id": "uuid",
        "attribution": {
            "verdict": "ai|human|uncertain",
            "combined_ai_score": 0.0,
            "confidence": 0.0
        },
        "signals": {
            "llm": { "ai_score": 0.0, "rationale": "..." },
            "stylometry": {
                "ai_score": 0.0,
                "metrics": {
                    "burstiness": 0.0,
                    "type_token_ratio": 0.0,
                    "punctuation_density": 0.0
                }
            }
        },
        "label": {
            "variant": "high_confidence_ai",
            "title": "🤖 Likely AI-generated",
            "body": "..."
        },
        "status": "classified",
        "timestamp": "ISO-8601"
    }
    ```
- **Errors:** `400` (missing/empty/too-short text), `429` (rate limit exceeded).

### `POST /appeal`

Contest a classification.

- **Accepts:** `{ "content_id": "uuid", "creator_reasoning": "<why the creator thinks it's wrong>" }`
- **Returns `200`:** `{ "content_id": "uuid", "appeal_id": "uuid", "status": "under_review", "timestamp": "ISO-8601" }`
- **Errors:** `400` (missing reasoning), `404` (unknown content_id).

### `GET /log`

Return the structured audit log (most recent first).

- **Returns `200`:** `{ "entries": [ { "type": "classification|appeal", "content_id": "...", "verdict": "...", "confidence": 0.0, "signals": {...}, "creator_reasoning": "...", "timestamp": "..." }, ... ] }`

### `GET /content/<content_id>` _(supporting)_

Return current status + last decision for one submission.

### `GET /health` _(supporting)_

Liveness check → `{ "status": "ok" }`.

## Appeals workflow

- **Who can appeal:** the creator of a submission — anyone holding the `content_id`
  returned by `/submit`. (On a real platform this would be gated by the creator's
  authenticated session; for this project the `content_id` is the handle.) An appeal can
  be filed against any classification, but it is most meaningful for an `ai` or
  `uncertain` verdict the creator disputes.
- **What they provide:** the `content_id` and free-text `creator_reasoning` explaining
  why they believe the verdict is wrong (e.g., "I wrote this by hand over three weeks;
  the formal tone is just how I write"). Reasoning is **required** — an empty appeal is
  rejected with `400`.
- **What the system does on receipt:**
    1. Looks up the submission in the content store (`404` if unknown).
    2. Flips its status from `classified` → **`under_review`**.
    3. Appends an **appeal entry** to the audit log, carrying the `content_id`, the
       creator's reasoning, a new `appeal_id`, and a timestamp — recorded **alongside**
       the original classification entry so the two are visibly linked.
    4. Returns `{ content_id, appeal_id, status: "under_review", timestamp }`.
  No automated re-classification occurs; resolution is a human decision.
- **What a human reviewer sees in the appeal queue:** querying `GET /log` (or a future
  moderation view) surfaces each appeal next to the decision it contests, so a reviewer
  sees, in one place: the original verdict and confidence, both individual signal scores
  (LLM rationale + stylometry metrics), the label text shown to the reader, the creator's
  reasoning, and timestamps for both events. That is enough context to uphold or overturn
  the verdict without re-running the pipeline.

## False-positive scenario (traced end to end)

**Scenario:** A non-native English speaker submits a heartfelt but grammatically clean,
formally structured personal essay. Its uniform structure makes it look statistically
"AI-like."

1. **`POST /submit`** → passes the rate limiter.
2. **Signal 2 (stylometry)** sees low burstiness and even punctuation → high
   `stylo_ai_score` (≈ 0.75). This is the blind spot in action: structure alone is
   misleading here.
3. **Signal 1 (LLM)** detects genuine personal specifics and emotional texture → lower
   `llm_ai_score` (≈ 0.40).
4. **Confidence scorer:** the signals **disagree** (`agreement` ≈ 0.65), and
   `combined_ai_score` ≈ `0.65*0.40 + 0.35*0.75 ≈ 0.52` — right in the middle.
   `confidence` is therefore **low**.
5. **Label builder:** because `confidence < 0.60` and the score sits in the middle band,
   the verdict is **`uncertain`**, not `ai`. The reader sees _"❓ Origin unclear … we're
   not making a call,"_ — **not** a false accusation.
6. **Appeal:** if the creator still objects, **`POST /appeal`** with their reasoning flips
   status to `under_review` and logs the appeal beside the original decision for a human
   moderator.

This is the asymmetry-of-harm principle working as intended: the system's design makes a
confident _false_ "AI" verdict structurally hard to reach, and provides a fallback path
(appeal) when it's still wrong.

### Other anticipated edge cases

- **Very short text (e.g., a 2-line poem):** stylometry is statistically unreliable on
  short inputs. Plan: enforce a minimum length (`400` on too-short text) and/or
  down-weight stylometry so the LLM dominates; surface low confidence rather than a
  confident call.
- **Groq API unavailable / errors:** the LLM signal may fail. Plan: degrade gracefully —
  fall back to a stylometry-only score with a capped/penalized confidence, and note in
  the audit log that only one signal was available, rather than failing the request.

## Stretch features (planned before building, per the brief)

All four stretch features were implemented. This section is the pre-work plan; the
README documents what was actually built.

### 1. Ensemble detection (3rd signal + voting)

- **Plan:** add a third, genuinely distinct signal — a **lexical "AI-tell" detector**
  (pure Python) that measures the *density of formulaic boilerplate phrases* ("it is
  important to note", "furthermore", "delve into", …). It is distinct from stylometry
  (structure) and the LLM (meaning): it targets *phrasing*.
- **Weighting / conflict resolution:** weighted ensemble — LLM 0.50, stylometry 0.30,
  lexical 0.20 (trust order). Conflicts are resolved by (a) the weighted vote, (b)
  down-weighting any signal that self-reports unreliable, and (c) a **dispersion penalty**:
  when the three signals disagree (wide score spread), confidence drops and the verdict
  falls back to `uncertain`. Individual signal scores are returned alongside the ensemble
  result.
- **Edge case:** a signal that flips the majority (e.g., lexical sees no tells in a formal
  human essay while the others lean AI) should *increase* dispersion → lower confidence,
  not silently get outvoted. This is desirable: it strengthens the false-positive guard.

### 2. Provenance certificate ("Verified Human")

- **Plan:** a creator earns a certificate via a **two-step challenge–response**: request a
  one-time pass-phrase (`GET /certify/challenge`), then echo it back with an authorship
  attestation (`POST /certify`). The certificate attests the *creator*, not the *text*.
- **Display:** a certified creator's submissions carry a `✅ Verified Human` badge that is
  **distinct from** the automated detection label and shown *alongside* it, so a verified
  credential is never confused with an automated estimate.
- **Edge case:** a challenge is single-use and tied to one creator id; replay or a
  mismatched phrase is rejected.

### 3. Analytics dashboard

- **Plan:** a `GET /analytics` endpoint (+ a small `/dashboard` HTML view) computing
  metrics from the audit log: detection pattern (AI/human/uncertain counts + ratio),
  appeal rate, average confidence, uncertain rate, and verified-creator count.

### 4. Multi-modal support (images)

- **Plan:** `POST /submit` accepts `content_type: "image"` with structured `metadata`
  and/or a `caption`. The pipeline swaps in image-appropriate signals: a **metadata
  provenance** check (AI-generator tags, C2PA flags, camera-EXIF presence) as the
  structural analogue of stylometry, plus the **LLM applied to the caption** as the
  semantic analogue. Both feed the same ensemble scorer, so confidence and labels work
  identically across modalities.
- **Edge case:** an image with neither metadata nor caption is rejected (`400`); a bare
  "no EXIF" guess is marked low-reliability so it can't drive a confident verdict alone.
